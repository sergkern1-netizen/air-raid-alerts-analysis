## 1. Аналіз собраних даних

### 1.1 Джерела даних

Проект використовує **два основні джерела** даних про повітряні тревоги в Україні:

## 4. Проблеми при збиранні та обробці даних

### 4.1 Проблема 1: Розбіжності між джерелами (Correlation: -0.30)

**Опис:** GitHub та Kaggle дані показують **від'ємну кореляцію** (-0.30) на денному рівні, що здивувало.

**Причина:** 
- GitHub дані – офіційні дані з державних датчиків сирен
- Kaggle дані – дані, зібрані через парсинг Telegram-каналів (громадські повідомлення)
- Telegram дані часто затримуються на 1-2 дні, тому денні числа розходяться

**Рішення:**
- Об'єднували дані через **union** (не intersection)
- Дедублікували записи для уникнення подвійних рахунків
- Використовували GitHub як первинне джерело (державні дані більш надійні)

**Наслідок:** Фіналізовано 273,270 унікальних записів (більшість з GitHub).

### 4.2 Проблема 2: Пропущені дані в деяких регіонах

**Опис:** Деякі регіони в Kaggle-наборі мають менше записів або їх немає взагалі на ранніх етапах конфлікту.

| Період | Покриття |
|--------|----------|
| 2022 Q1-Q2 | ~40% регіонів (война до почала, слабка система моніторингу) |
| 2022 Q3-Q4 | ~80% регіонів (система розширена) |
| 2023+ | 100% регіонів (стабільна система) |

**Рішення:**
- Розраховували статистику окремо для периодів з повним покриттям
- Не інтерполювали пропущені дані (залишили їх як є)
- В моделюванні використовували дані з 2023+ (100% покриття)

**Наслідок:** Надійна статистика по регіонам з 2023+ років.

### 4.3 Проблема 3: Нестаціонарність часового ряду

**Опис:** Часовий ряд **НЕ стаціонарний** (середнє значення змінюється в часі, немає сталого рівня).

**Тест:** Augmented Dickey-Fuller (ADF) p-value = 0.581 > 0.05  
**Висновок:** Ряд має тренд, дисперсія змінюється.

**Причина:** Реальна ескалація конфлікту з часом!

**Рішення:**
- Для моделювання використовували **диференціацію** (перші різниці)
- Використовували **адаптивні моделі** (ExponentialSmoothing, LSTM) замість класичних (ARIMA)
- Моделі, що враховують тренди, дають краще прогнозування

**Наслідок:** ExponentialSmoothing показав найкращу точність (MAPE 75.42%).

### 4.4 Проблема 4: Викиди та екстремальні значення

**Опис:** 177 днів (11.3%) мають **екстремально високі значення** (>400 тревог на день).

**Приклади:**
- 2024-05-12: 1,004 тревоги (максимум)
- 2025-12-15: 876 тревог (майже максимум)
- Q4 2025: Багато днів > 500 тревог

**Тест:** Shapiro-Wilk p-value = 9.07e-44 → **розподіл НЕ нормальний**, права хвіст.

**Рішення:**
- **НЕ видаляли** викиди (вони реальні, важливі для прогнозування)
- Використовували **robust регресію** в моделях
- Стратифікували модель на два типи днів: нормальні (<250 тревог) і критичні (>250)

**Наслідок:** Моделі враховують обидва режими роботи системи.

### 4.5 Проблема 5: Дисбаланс між периодами навчання та тестування

**Опис:** 
- **Тренувальний набір** (2023-03-12): середнє = 14,722 тревог/день, σ = 65,886
- **Тестовий набір** (2025-03-13 по теперішнє): середнє = 44,179 тревог/день, σ = 24,795
- **Різниця:** 3x вищі значення в тестовому наборі!

**Причина:** Істинна ескалація конфлікту за час навчання-тестування.

**Вплив на точність:** MAPE природно вище (~75%+) через розпреділ-shift.

**Рішення:**
- Прийняли MAPE 75%+ як **розумний базовий рівень** (не ARIMA-проблема, а реальна даних-проблема)
- Рекомендуємо **моніторинг MAPE в реальному часі** для виявлення подальшої зміни розподілу
- При MAPE > 90% – перенавчити модель на новіших даних

**Наслідок:** Розуміння обмежень моделі в умовах ескалації.

### 4.6 Проблема 6: Потребу у Real-Time оновленнях

**Опис:** Дані до 2026-06-24, але конфлікт тривала, нові дані надходять щодня.

**Виклик:** Модель навчена на даних, які 3+ місяці на це старіші.

**Рішення, запропоноване:**
- Встановити **pipeline для щомісячного переобучання** моделі
- Використовувати **API для real-time інгесту** даних (коли Alerts-ua-py API стане доступним)
- Налагодити **моніторинг якості** (якщо MAPE > 30% – Alert!)

**Наслідок:** Production-ready архітектура для майбутніх версій.
